# Sell-side Polish Checklist

Use this near the end of a report project, especially for professional or external-facing deliverables.

## Final Identity

The final artifact should read as a complete industry/company research report, not as:

- research memo
- evidence audit
- process record
- literature review
- source summary

The report should keep source discipline in the background while foregrounding the industry/company logic. The evidence audit trail should live in supporting files, especially `outputs/02-evidence-book.md` and `final/claim-source-map.md`, not in long source-heavy paragraphs inside the main report.

## Structure

- Does the report answer one central question?
- Does every chapter support the central thesis?
- Is the structure organized by industry/company logic rather than source order?
- Are repeated sections merged?
- Are cases used as support rather than the main spine?
- Are risks and counterarguments placed where they improve judgment rather than interrupt every paragraph?

## “Why” Logic

Prefer explanatory headings:

- Weak: `渠道介绍`
- Strong: `为什么渠道效率决定商用清洁机器人的盈利弹性？`

- Weak: `成本结构`
- Strong: `为什么核心零部件降价会放大利润修复？`

- Weak: `海外市场`
- Strong: `为什么本地服务网络决定工业传感器出海上限？`

Rewrite encyclopedia-style sections:

- `渠道模式介绍` → `为什么渠道效率决定商用清洁机器人的盈利弹性？`
- `设备租赁业务介绍` → `为什么客户回本周期决定低温仓储设备的渗透速度？`
- `成本结构有哪些` → `为什么核心零部件降价会放大利润修复？`
- `海外市场包括什么` → `为什么本地服务网络决定工业传感器出海上限？`

## Chapter Takeaways

Add a short `本章小结` to serious reports. It should state the point, not summarize every paragraph.

Good pattern:

`本章小结：本轮商用清洁机器人修复的核心，不只是需求回暖，而是渠道效率、零部件成本改善和产品结构升级共同推动利润弹性释放。后续研究应从单纯销量跟踪，转向对价格、费用率和渠道质量的综合判断。`

## Visuals

Prefer:

- core thesis framework
- value chain diagram
- commercialization path
- global expansion path
- growth driver bridge
- risk transmission map
- segment comparison table with short cells
- case box
- chapter conclusion box

For serious reports, create at least one core framework visual near the beginning. It should explain the report’s logic, not decorate it.

Avoid:

- wide tables with long Chinese prose
- dense source audit tables in the main body
- diagrams that merely decorate rather than explain logic

## Risk Treatment

Good risk writing explains how the thesis could fail:

- trigger signal
- affected mechanism
- expected impact
- monitoring indicator
- why it matters to the central conclusion

Avoid generic endings such as only listing “政策风险、市场竞争风险、技术风险”. Tie each risk back to the report’s main logic.

## Language

Make the prose sound like a thoughtful analyst, not a process log.

Do:

- lead with judgment
- explain mechanism
- use concise transitions
- write limitations once, in the right place

Avoid overusing:

- “当前证据强度”
- “不能外推”
- “不构成高置信事实”
- “需要进一步验证”

These belong in methods, risks, limitations, or footnotes unless central to the argument.

## Final Gate

Before delivery, check:

- central thesis is clear within the first page
- each chapter has a reason to exist
- repeated chapters or duplicated themes have been merged
- evidence supports key claims
- material claims are traceable in `final/claim-source-map.md`
- risks and limitations are centralized
- counterarguments and failure conditions are addressed
- cases are supporting boxes rather than the spine
- chapter takeaways are present where useful
- headings explain “why,” not only “what”
- document is editable
- no wide-table layout bug
- no unsupported investment recommendation
